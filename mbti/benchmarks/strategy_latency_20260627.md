# explain mode · 응답 전송 전략별 체감 LLM 시간 (2026-06-27)

스크립트: `benchmarks/run_strategy_bench.py` (explain · K=2 · API 백엔드 · prefetch dwell=4s).
체감 = 유저가 답을 받기까지 느끼는 LLM 시간(ms). **작을수록 빠름.**

| 시나리오 | 기존 v1 (Sonnet·비스트림·2000) | v2 비스트림 (Haiku·1200) | **v2 스트림 (첫 토큰)** | v2 prefetch (dwell 4s) |
|---|---|---|---|---|
| 로그인 에러 | 9146 | 7470 | **684** | 1568 |
| 교통 안내문 | 11099 | 6428 | **688** | 1873 |
| 카페 메뉴 | 9058 | 4931 | **756** | 947 |
| 회원가입 폼 | 7970 | 7516 | **672** | 2993 |
| **평균** | **9318** | **6587** | **700** | **1845** |

## 전략 정의
- **기존 v1**: Sonnet · 비스트리밍 · 2000tok → 완성까지 대기(ms_total)
- **v2 비스트림**: Haiku · 비스트리밍 · 1200tok → 완성까지 대기
- **v2 스트림**: Haiku · 스트리밍 → 첫 토큰(TTFT) = 반응 시작 시점
- **v2 prefetch**: full을 미리 생성, "더 자세히" 클릭 시 체감 = max(0, full_total − dwell)
  (dwell = brief 읽는 시간, 4s 가정)

## 결론
- 기존 v1(~9.3s) → v2 비스트림(~6.6s): 모델 교체만으로 단축, 그래도 길다.
- **v2 스트림(~0.7s 첫 토큰)이 압도적** — 4시나리오 모두 일관.
- v2 prefetch(~1.8s): 비스트림보단 훨 낫지만 **스트림엔 못 미침**(full이 4초에 안 끝나 남은 시간 대기) + full 항상 호출 비용.
- → **권장: v2 + 스트리밍.** prefetch는 비추(스트리밍 대비 느리고 비용↑).

## 네이밍 메모
prefetch는 **새 API 버전(v3)이 아님** — v2 엔드포인트 위의 클라이언트 전송 전략(미리 호출).
DB `api_version`(1/2=엔드포인트)과 구분. v3는 진짜 다른 서버 계약(새 모델/프로바이더/엔드포인트)에 사용.
