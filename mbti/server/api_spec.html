연계 흐름

모든 요청/응답은 application/json. 시간은 UTC ISO-8601 문자열. CORS는 동일 출처(localhost) 기준. 아래 코드블록은 우상단 복사 버튼으로 바로 가져갈 수 있어요. (<base64> 부분만 실제 이미지로 채우세요 — 변환법은 이미지 → base64)

⚡ 시나리오별 요청 — 복사해서 바로 쓰기

프런트가 가장 자주 쓰는 호출들. 전부 POST · application/json. 모델은 서버가 자동 선택(F형→Claude, T형→GPT mini).

이미지 → base64 변환 (image_b64 만들기)

이미지는 파일 첨부(multipart)가 아니라 JSON 본문의 image_b64 필드에 base64 텍스트로 담아 보냅니다. Content-Type은 application/json. 주의: data:image/png;base64, 접두사는 반드시 제거(비전 API가 거부), media_type은 실제 타입과 일치, 원본 ≤ 5MB(base64는 ~33% 커짐).

브라우저 (가장 흔함 — <input type="file">)

function fileToBase64(file) {
  return new Promise((resolve, reject) => {
    const fr = new FileReader();
    fr.onload = () => resolve(fr.result.split(",")[1]); // "data:...;base64," 제거
    fr.onerror = reject;
    fr.readAsDataURL(file);
  });
}
const file = input.files[0];
const image_b64 = await fileToBase64(file);
const media_type = file.type;            // "image/png" 등

브라우저 — 5MB 초과 시 리사이즈+압축

async function shrinkToBase64(file, max = 1280, q = 0.8) {
  const img = await createImageBitmap(file);
  const s = Math.min(1, max / Math.max(img.width, img.height));
  const c = document.createElement("canvas");
  c.width = img.width * s; c.height = img.height * s;
  c.getContext("2d").drawImage(img, 0, 0, c.width, c.height);
  const url = c.toDataURL("image/jpeg", q);
  return { image_b64: url.split(",")[1], media_type: "image/jpeg" };
}

conversation_id vs parent_id (대화/맥락) — 중요

둘은 역할이 완전히 다릅니다. 한쪽은 "묶기(분류·조회)", 다른쪽은 "AI 맥락 주입"입니다.

conversation_idparent_id
뜻어느 대화 묶음(스레드)에 속하나어떤 메시지에 대한 답글(reply)인가
비유채팅방 ID그 방에서 특정 메시지에 "답장"
값UUID 문자열 (서버 발급)메시지 id(int) 또는 null
AI 답변에 영향?❌ 없음 — 저장/조회용 라벨일 뿐✅ 있음 — 부모의 text/result를 프롬프트에 끼움
용도/conversation/{id} 로 함께 조회, 세션 유지AI가 "이전 대화를 기억"하게 만듦
핵심: conversation_id 는 메시지를 묶기만 하고 AI 호출엔 안 들어갑니다. 실제로 이전 맥락을 AI에게 주입하는 건 parent_id 입니다 — 서버가 그 부모의 text·result를 꺼내 프롬프트 앞에 아래처럼 붙입니다:
[이전 대화]
요청: {부모 메시지의 text}
답변: {부모 메시지의 result}
위 맥락을 참고해서 답해 주세요.
예시
대화방 "abc" (conversation_id)
 ├─ id=1  "이 표지판 번역해줘"          parent_id=null   // 최초
 ├─ id=2  "더 자세히"                  parent_id=1      // AI가 id=1 내용을 보고 답함
 └─ id=3  "다른 표지판이야"            parent_id=null   // 같은 방이지만 맥락 주입 X

→ id 1·2·3 은 같은 conversation_id 라 /conversation/abc 로 한꺼번에 조회됨.
→ id=2 만 parent_id=1 이라 AI가 1번을 참고. id=3 은 같은 방이어도 독립적으로 답함.
프론트 사용 규칙
상황conversation_idparent_id
첫 요청보내지 않음 (서버가 발급)null
직전 답에 이어가기받은 값 그대로직전 응답의 id
같은 방, 새 주제받은 값 그대로null (맥락 주입 안 함)
⚠️ parent_id 가 존재하지 않는 id면 404. 부모 메시지를 DELETE 하면 그 자식들의 parent_id 는 자동으로 null 로 정리됩니다(맥락만 끊기고 메시지는 유지).

POST /analyze/stream SSE 실시간
요청 바디는 /analyze와 동일. 응답은 Server-Sent Events(text/event-stream)로 단계·토큰·완료를 실시간 전송 → 로딩 대기감↓. 각 줄은 data: {json}\n\n.

요청 바디는 v1과 동일(+ detail). v1·v2 둘 다 살아있어 A/B 테스트 가능. 응답에 version 포함.

v1과 같은 로직, 백엔드 노브만 다름 — 측정상 translate ~7s→~2s, explain ~14s→~2s.

길게 보기: detail:"full"로 한 번에, 또는 짧게 받고 "더 자세히" = parent_id 붙인 reply(점진적 확장). 둘 다 지원.
// 응답에 version 추가
{ "id":7, "result":"...", "version":2, "detail":"brief", "ocr_source":"client", ... }

prefetch — "더 자세히" 즉시화 (프론트 패턴, 서버 변경 없음)

brief를 보여주고 유저가 읽는 동안 full을 백그라운드로 미리 호출해 두면, "더 자세히" 클릭 시 즉시 제공된다. 서버는 그대로(기존 /v2/analyze 사용) — 프론트에서 일반 fetch 한 번 더가 전부(스트리밍 SSE 불필요).

핵심: "더 자세히"는 클릭 시점에 API를 호출하지 않는다. full 요청(②)은 brief 직후 백그라운드로 미리 나가고, 클릭은 그 결과(Promise)를 받을 뿐.

시간 ───────────────────────────────────────────────▶

[분석 요청]
  ├─① POST /v2/analyze  detail:"brief"
  │     └◀ {id, conversation_id, result}   (~2s)  → 화면 표시, 유저 읽기 📖
  └─② POST /v2/analyze  detail:"full", parent_id, conversation_id   ← brief 직후 자동·백그라운드
        └◀ {id, parent_id, result}  (~5s, 유저 모름)  → fullPromise 보관

["더 자세히" 클릭]  ❌ 새 호출 없음 →  await fullPromise → 즉시 ②결과 표시 ⚡
인수① brief (첫 요청)② full (= "더 자세히", 백그라운드)
persona / mode / image_b64 / ocr_text지정①과 동일
detail"brief""full" ← 차이
conversation_id없음(첫 요청)brief.conversation_id (①응답)
parent_id없음brief.id (①응답 — 이어받기)
즉 "더 자세히" = 호출 ②(detail:"full" + parent_id로 brief에 이어붙인 reply). 클릭 전에 미리 보낼 뿐. API는 /v2/analyze 하나, brief·full 차이는 detail뿐.
let fullPromise = null, fullAbort = null;

// 1) brief 표시 후, full을 곧바로 백그라운드 호출 (독립 — brief 안 기다림)
async function analyze(req) {
  const brief = await postJSON("/v2/analyze", { ...req, detail: "brief" });
  render(brief);                                   // 유저가 brief 읽기 시작
  fullAbort = new AbortController();
  fullPromise = postJSON("/v2/analyze",
      { ...req, detail: "full", conversation_id: brief.conversation_id, parent_id: brief.id },
      { signal: fullAbort.signal }).catch(() => null);
}
// 2) "더 자세히" 클릭 → 미리 만든 것 즉시(또는 남은 시간만)
async function onMoreDetail() { const f = await fullPromise; if (f) render(f); }
// 3) 화면 이탈 시 prefetch 취소 → 낭비 방지
function onLeave() { fullAbort?.abort(); }
비용 주의: "더 자세히"를 안 눌러도 full을 매번 호출하므로 비용이 듦 → onLeave 취소(+선택적 결과 캐시) 권장. 측정/비교는 /sample-test(localhost)에서 가능. 콘솔(/)의 "더 자세히"가 이 패턴의 동작 레퍼런스.

기본 모드 & 능동 제안 — 프런트 가이드

MBTI 스타일이 답을 해칠 수 있을 때 담백한 "기본 모드"로 전환하는 흐름. 프런트가 할 일은 토글 1개 + boolean 1개가 전부.

필드방향프런트가 할 일
styled요청(기본 true)상시 토글 노출. 유저가 "담백하게" 켜면 styled:false로 보냄
styled응답(에코)지금 답이 어느 모드인지 표시(선택)
suggest_plain응답true면 기본 모드 토글을 강조(배지/하이라이트). 강제 아님 — 유저가 선택
// 흐름
1) 평소 요청 (styled 생략 = true)
   → { result, styled:true, suggest_plain }     // 마커는 서버가 떼서 result엔 없음
2) suggest_plain === true 면:
   → 기본 모드 토글을 "강조"만 (💡 이 답은 기본 모드가 나을 수 있어요)
3) 유저가 토글 ON →  같은 요청을 styled:false 로 재요청
   → { result(담백), styled:false }              // 보통 suggest_plain 안 옴

// 코드
const r = await postJSON("/v2/analyze", { ...req, styled });   // styled 기본 true
render(r.result);
if (r.suggest_plain) highlightPlainToggle();                   // 강조만(강제 X)
// 토글 클릭 핸들러: styled=false 로 같은 요청 재전송 → 담백 답
스트리밍(/v2/analyze/stream): done 전에 {"type":"suggest_plain"} 이벤트가 올 수 있음 → 그때 토글 강조. 요점: 토글은 항상 노출(유저가 언제든 전환), suggest_plain은 "지금 권장"을 강조하는 신호일 뿐. 내부 마커는 서버가 제거하므로 result엔 안 보임. 동작 레퍼런스 = 콘솔(/)의 스타일 토글 + 💡 안내.

연계 흐름

1) (선택) 퀴즈로 MBTI 진단
   POST /quiz/start → POST /quiz/answer ×4 → { result:"ENFP" }

2) 분석 요청 (최초)
   POST /analyze { persona:"ENFP", mode, text, image_b64 }
        → { id:1, conversation_id:"abc", parent_id:null, result }

3) 이어서 reply (이전 맥락 반영)
   POST /analyze { conversation_id:"abc", parent_id:1, ... }
        → 서버가 1번 text/result를 프롬프트에 끼움 → { id:2, parent_id:1 }

4) 다시 보기
   GET /conversation/abc   ·   GET /users/minsu/messages
대화 유지: 첫 응답의 id·conversation_id를 프론트가 들고 다니며 후속 요청에 전달.
